arpit-blog — Technical Overview

A static, content-first personal site built with Astro. Three writing formats (notes, articles, links) live as Markdown/MDX files in the repo; Astro reads them at build time, validates their frontmatter, and generates a fully static HTML site with paginated feeds, tag pages, an archive, RSS, and client-side search.

Astro 6 MDX Pagefind TypeScript Zero JS framework Netlify

1. What this site is

Three content types coexist on the homepage feed, each with its own section as well:

  • Notes — short, dateless-title posts (Markdown).
  • Articles — longer, titled posts (Markdown/MDX), can have an updatedDate.
  • Links — bookmark-style entries that point to an external URL and may credit a via source.

Plus two book galleries (Library = read, Antilibrary = owned-unread) sourced from a single books.json file.

2. What Astro does for you

Astro is a static-site generator with a component model that looks like JSX but runs entirely at build time. The mental model:

  • File-based routing. Anything in src/pages/ becomes a URL. about.astro → /about/. Brackets in the filename mean dynamic segments: [slug].astro, [...page].astro (rest).
  • Content collections. A typed wrapper around a folder of Markdown/MDX. You declare a Zod schema for frontmatter once; Astro validates every file and gives you a typed API (getCollection('notes')) to read them.
  • .astro components. Two parts separated by --- fences: a JS/TS frontmatter section (runs at build time on the server) and an HTML-with-expressions template. There is no client runtime unless you add <script> tags.
  • Zero JS by default. The output is plain HTML + CSS. The few interactive widgets here (copy-to-clipboard, search, Mastodon share, book filters) are tiny vanilla JS files in src/scripts/ loaded via <script src>.
  • Static build. astro build writes everything to dist/. There is no server running in production — Netlify just serves files.

5. Content collections

Defined in src/content.config.ts. Each collection has a loader (where files are) and a schema (what frontmatter must look like). All three text collections share a base shape; books is a special case backed by a single JSON file.

notes

Glob: src/data/notes/**/*.{md,mdx}

Schema: rssSchema + pubDate, tags[], optional draft. No required title — notes are dateless-titled.

articles

Glob: src/data/articles/**/*.{md,mdx}

Adds required title and optional updatedDate.

links

Glob: src/data/links/**/*.{md,mdx}

Adds required title, link (external URL), and optional via: { url, label }.

books

Loader: file('src/data/books.json') — every object in the JSON array becomes one entry.

Fields: id, title, subtitle?, author, cover, tags[], status, dominantColor. status is an enum: unread | read | reading.

Two important details about the schemas:

  • pubDate accepts string/number/Date and is transformed into a JS Date object. Downstream code can just call .getFullYear() without parsing.
  • Each schema .transform()s the entry to also expose categories (a copy of tags) so RSS items have what they need with no extra mapping.

What an entry file looks like

---
pubDate: "2026-03-11T18:52+0530"
tags:
  - "anchor-positioning"
  - "css"
---

6. Data flow — from .md to rendered page

Two small utilities mediate between raw collections and the pages that consume them.

loadAndFormatCollection(name)

In src/utils/collection.ts. For one collection it:

  1. Calls getCollection(name) (Astro’s built-in).
  2. Filters out draft: true entries in production only.
  3. For each entry, calls render(entry) to get a <Content /> component (compiled MDX/MD).
  4. Computes relativeURL from YYYY/MM/slug/ and an absoluteURL like /notes/2026/03/foo/.
  5. Sorts newest first by pubDate.

The return type is EnhancedEntry<T>[] — see src/types/entries.ts: a CollectionEntry plus the three injected fields (Content, relativeURL, absoluteURL).

loadAllEntries()

In src/utils/loadAllEntries.ts. Loads notes, articles, links in parallel, concatenates them, and sorts the union newest-first. This is what powers anything that mixes types: the homepage feed, the archive, tag pages, and the combined RSS feed.

.md / .mdx files │ ▼ getCollection() ← Astro reads & Zod-validates frontmatter │ ▼ loadAndFormatCollection() ← drafts filtered, render() called, │ URLs computed, sorted by date ▼ loadAllEntries() ← merges notes + articles + links │ ▼ page .astro file ← getStaticPaths() or top-of-file await │ ▼ Layout (FeedPage / EntryPage) │ ▼ built HTML in dist/

7. Routing

Pages fall into three shapes.

Static pages

Plain .astro files with no params: archive.astro, tags/index.astro, search/index.astro, library/index.astro, antilibrary/index.astro. Each does its own await at the top of the frontmatter.

Paginated feeds — [...page].astro

The rest-bracket name is an Astro convention: page 1 is at the parent URL, subsequent pages get appended as /2/, /3/. The route exports getStaticPaths and uses the paginate helper Astro injects: 

export const getStaticPaths = (async ({ paginate }) => {
  const allEntries = await loadAllEntries();
  return paginate(allEntries, { pageSize: 20 });
});

Used for: /, /notes/, /articles/, /links/.

Dynamic entry pages — [...slug].astro

Per-collection. getStaticPaths maps every entry to a route whose slug param is its precomputed relativeURL (YYYY/MM/file-id/): 

export async function getStaticPaths() {
  const posts = await loadAndFormatCollection('notes');
  return posts.map((post) => ({
    params: { slug: post.relativeURL },
    props: { entry: post },
  }));
}

The page just renders <EntryPage entry={entry} />.

8. Layouts & components

Four layouts compose the site. Each page picks one:

Layout Used by Renders
BaseLayout everything <html> shell + BaseHead + SkipLink + <slot />
FeedPage notes, articles, links list pages banner, header, paginated entries, sidebar with “currently reading”
EntryPage single note/article/link pages heading, body, pub/updated dates, tags, share actions, back link
BookGallery library + antilibrary filterable book grid, view-transition names per cover

BaseHead handles all the head-tag plumbing in one place: title, OG/Twitter meta, canonical URL, fonts (preload + variable font via astro-font), CSS, favicons (an emoji SVG in dev, real files in prod), the four RSS <link rel="alternate">s, and the tiny inline script that swaps no-js → js on <html>.

Component highlights:

  • EntryHeading picks the right element/format per collection type.
  • EntryList + PaginatedEntries render lists with pagination controls.
  • SiteNav computes aria-current="page" vs "true" for active vs ancestor routes.
  • EntryActions, CopyToClipboard, MastodonShare, BackLink — small widgets, each paired with a tiny script in src/scripts/.
  • CurrentlyReading reads books entries where status === 'reading'.
Type guard trick worth noting in EntryPage.astro: because entry.data is a union of three schemas, accessing updatedDate or via uses 'updatedDate' in entry.data as a TypeScript narrowing guard — clean and statically checked.

9. RSS feeds

Four feeds are generated, all as Astro endpoints (.js files in pages/ that export GET):

  • /rss.xml — everything
  • /notes/rss.xml, /articles/rss.xml, /links/rss.xml

The interesting bit is in src/utils/rss.ts: feeds need full HTML, but render(entry) gives you an Astro component, not a string. The util spins up an experimental_AstroContainer with the MDX renderer loaded, renders each Content component to a string, then rewrites every relative src/href to absolute using baseUrl. Per-collection branching maps notes to description, articles/links to content, and links use the external URL as their link.

10. Search — Pagefind

Pagefind is a static search indexer. After astro build, the astro-pagefind integration crawls dist/ and writes a /pagefind/ directory of pre-built index shards + a JS client. At runtime, the browser fetches just the shards it needs.

Two pieces wire this together:

  • In EntryPage.astro, the body wrapper has data-pagefind-body, data-pagefind-meta="date:…", and data-pagefind-filter="collection:…". That tells Pagefind what to index and how to facet it.
  • /search/ renders the UI and loads /src/scripts/search.js, which imports /pagefind/pagefind.js at runtime. The Vite config marks that path as external so Rollup doesn’t try to bundle it (it only exists after the post-build step).

Search is disabled in npm run dev because the index doesn’t exist there — the search page shows a friendly message instead.

11. Books pipeline

Books are a JSON-backed collection — fast to edit, no per-file frontmatter overhead. Each book has a cover image and a precomputed dominantColor used for the wrapper background.

That color isn’t computed at build time. There’s a separate Node script: scripts/update-book-colors.mjs, run via npm run books:colors. It reads each cover with sharp, derives a dominant color, and writes it back into books.json. So the build stays fast; new books just need the script run once.

BookGallery.astro filters by status (passed as a prop — ['read'] for library, ['unread'] for antilibrary), assigns a unique view-transition-name per cover, and applies a small random rotation for visual character.

12. Styling

Plain CSS, hand-rolled, no utility framework. The folder structure under src/css/ follows a CUBE-like organisation:

  • reset.css, global.css, style.css — root-level entries
  • theme/ — design tokens (colors, type scale, space scale)
  • blocks/ — component-scoped styles
  • composition/ — layout primitives (stack, cluster, center, flow…)
  • exceptions/ — one-off overrides
  • utilities/ — single-purpose helpers

Fluid type and spacing come from Utopia via postcss-utopia. postcss-preset-env handles future syntax for the browserslist target baseline widely available. Shiki provides syntax highlighting for code blocks using the css-variables theme — so your CSS controls the palette.

13. Build & deploy

Command What it does
npm run dev Local dev server at localhost:4321 with HMR. Drafts visible. Search disabled.
npm run build astro check (type-check .astro + TS) then astro build → static dist/, drafts filtered, HTML minified, Pagefind index generated.
npm run preview Serves the dist/ output locally to sanity-check the production build.
npm run books:colors Recompute dominant colors for book covers.

Hosting: Netlify (the .netlify/ folder gives it away). It’s static hosting — Netlify builds the repo, serves dist/. No serverless functions, no edge logic.

14. Mental model recap

If you forget everything else, hold on to this:

  1. Content is files. Notes/articles/links are Markdown in src/data/. Books are one JSON file. There is no CMS, no database. Writing a new post = creating a file.
  2. Schemas validate frontmatter. If a date is missing or malformed, astro check fails. Frontmatter dates arrive in .astro code as real Date objects.
  3. Pages do the orchestration. Each route awaits the data it needs (via loadAllEntries or loadAndFormatCollection), and either renders directly or uses getStaticPaths to fan out into many static URLs.
  4. Build time, not request time. Everything you see in .astro frontmatter executes once during astro build. The browser only sees plain HTML + a few small vanilla JS files.
  5. Two interactive features need post-build steps. Pagefind indexes dist/ after build for search; book colors are pre-computed by a separate script when you add a book.

Adding a new note? Drop a Markdown file in src/data/notes/, fill in pubDate and tags, rebuild. The home feed, the notes feed, the relevant tag pages, the archive month, all four RSS feeds, and the search index all update automatically.
